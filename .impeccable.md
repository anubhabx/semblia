## Design Context

### Users
SaaS founders, indie developers, consultants, and freelancers who collect social proof for their products/services. Used during work hours (9–6), on desktop primarily, in well-lit office or home-office environments. Their job: quickly triage incoming testimonials, publish the good ones, embed them on their site.

### Brand Personality
**Measured. Credible. Warm.**
- NOT: flashy, dark-terminal aesthetic, data-dense dashboards, playful/bubbly
- The product handles trust signals — it must itself feel trustworthy
- Warm because testimonials are human; precise because the owner is a professional

### Aesthetic Direction
- **Theme**: Light. Daytime productivity tool. Warm linen/slate palette already established.
- **References**: Linear (tight sidebar, clean lists, keyboard-first), Vercel (project-centric, status badges, clean typography), Notion (structured but warm)
- **Anti-references**: Shadcn default demos (too cold), Framer-dark-glassmorphism, loud SaaS marketing dashboards
- **Typography**: Inter (existing) — use weight + size contrast more aggressively. Headings at 600–700, body at 400–450. Muted labels at 10–11px all-caps tracking.
- **Colour accent**: Single amber-sand brand colour (`--brand`). Used sparingly — pending badges, primary CTA, active nav. Everything else is warm neutrals.
- **Spacing**: Comfortable density. Not as tight as Linear, not as airy as Notion. ~16px base unit.

### Design Principles
1. **Project-first**: Every view centres a project. The global view is just a project selector — never a dashboard of aggregated vanity metrics.
2. **One thing at a time**: Each page has a single primary action. No competing CTAs.
3. **Earn trust through restraint**: No decorative flourishes. Every element must earn its place. Whitespace is intentional.
4. **Status is sacred**: Moderation status, verification badges, and publish state must be instantly scannable. Use shape + colour, never colour alone.
5. **Empty states teach**: When a user has no testimonials/widgets/keys, the empty state explains what the feature does and gives one action — not a wall of text.
6. **One system, derived everywhere**: There is a single source of truth for chrome. Compose the shared primitives in `apps/web_v2/components/shared` (`PageHeader`, `PageTabs`, `SectionNav`, `PageBody`, `EmptyKindPicker`, `SettingsSection`, ...) — never hand-roll a page's own header, tabs, or empty state. A section with >6 destinations uses the vertical `SectionNav` rail (config-driven), not a horizontal scroll-tab strip. If a primitive falls short, extend the primitive; do not fork a per-page copy. New surfaces must look native to the rest of the app, not like a different product.
